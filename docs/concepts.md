## Идея в одном абзаце

Основная концепция — разделить приложение на ортогональные оси:

- **идентичность объектов** (кто-кому-зависим, кто-сколько-живёт),
- **реактивное состояние** (что меняется),
- **серверное состояние** (что приходит из сети и кэшируется),
- **UI** (как всё отображается).

За первую отвечает [`@fozy-labs/simplest-di`](https://www.npmjs.com/package/@fozy-labs/simplest-di), за вторую и третью — [`@fozy-labs/rx-toolkit`](https://www.npmjs.com/package/@fozy-labs/rx-toolkit).
За четвёртую — React, который подписывается на сигналы и вызывает методы инжектированных сторов.
Сама бизнес-логика остаётся обычными классами и обычными функциями, не зависящими от React,
что даёт лёгкое тестирование, понятный поток данных и потенциальную переносимость на другие фреймворки.


## «Проблемы», которые мы решаем

Прежде чем описывать кирпичи стека, удобно зафиксировать болевые точки, под которые он спроектирован. Дальше каждая из них явно отыграется в разделе [«Что в итоге?»](#что-в-итоге).

### Вынос логики

Одна из тенденций в web-разработке — выносить бизнес-логику из компонентов в отдельные классы/сторы/сервисы (например, Vuex / Effector). Это даёт несколько преимуществ: **тестируемость**, **понятный поток данных** и **потенциальная переносимость между фреймворками**.

Важное уточнение: речь не просто про вынос в стор, как redux или zustand, а про описание реактивной логики, когда сервисы/сторы могут работать друг с другом, минуя React.

Но при этом возникает проблема: бизнес-логика становится оторвана от жизненного цикла компонентов. Например, если наша логика подписывается на сокет, то когда он должен открывать и закрывать соединение? Разные библиотеки решают это по-разному — например, предоставляют Gate-компонент или HOC-обёртки для подключения к жизненному циклу.

### Динамическое создание (и связывание)

Под эту «проблему» попадает:

1. **Контекстно-зависимый flow.** Например, если мы используем react-router, как связать логику с ID открытого поста.
2. **SSR.** Как при серверном рендере разделять состояния разных клиентов, не делая всё глобальным.

Часть библиотек закрывают эти проблемы собственным роутером, сложным механизмом форков или просто оставляют их пользователю.

### Query-менеджер

Killer-фича современной web-разработки — query-менеджер, который берёт на себя всю работу с серверным состоянием: кэширование, дедупликацию, оптимистичные апдейты, инвалидацию и т. д. Но большинство решений на рынке упираются в уже знакомые проблемы:

- состояние «прибито» к UI-слою;
- сложности с динамическим созданием и связыванием;
- нет реактивного ядра, к которому можно было бы подключить любую логику.

### Dev-tools

Понимание того, что происходит в приложении, — это половина успеха. Не все библиотеки предоставляют удобные инструменты для отладки.


## Что в итоге?

### Кирпичи стека

- **`Signal`** — базовый реактивный примитив. Снаружи похож на сигналы из SolidJS или Angular Signals, но под капотом — обычные `Observable` из RxJS ([подробнее про связь сигналов и RxJS — в доке пакета](https://github.com/fozy-labs/rx-toolkit/blob/main/docs/signals/README.md)). Не требует ручной подписки/отписки (кроме эффектов), поддерживает `compute`, batched updates, peek без подписки и интегрируется с DevTools. Сигналы могут создаваться динамически и удаляются из памяти стандартными механизмами GC.
- **`Api`** — реактивный query-менеджер. Закрывает классические задачи серверного состояния (кэш, дедупликация, инвалидация, оптимистичные апдейты), но в отличие от tanstack-query/swr живёт *вне* React: его параметры — это сигналы, и любая логика может на него подписаться или его триггернуть. Подробности — в [документации `query`](https://github.com/fozy-labs/rx-toolkit/blob/main/docs/query/README.md).
- **`SCOPE`** — концепт из [`@fozy-labs/simplest-di`](https://github.com/fozy-labs/simplest-di/blob/main/docs/concepts.md). Это контейнер для инстансов классов, который гарантирует их единственность там, где это нужно, и аккуратно их утилизирует. Скопы могут быть привязаны к жизненному циклу React-компонента, страницы или вообще быть глобальными. Они же — точка сцепки между React и бизнес-логикой (см. [react-integration](https://github.com/fozy-labs/simplest-di/blob/main/docs/react-integration.md) и [usage/react в rx-toolkit](https://github.com/fozy-labs/rx-toolkit/blob/main/docs/usage/react/README.md)).

### Что это даёт

Связка трёх кирпичей закрывает проблемы из предыдущего раздела:

- **Вынос логики** → бизнес-логика живёт в обычных классах с сигналами; её можно тестировать без React Testing Library, описывать декларативно и не размазывать по компонентам.
- **Динамическое создание/связывание** → SCOPED-классы и `inject(...)` дают единственность инстанса в нужной области, а скопы решают и SSR (по скопу на запрос), и привязку к контексту маршрута. Никто не мешает написать класс и создать его обычным `new` внутри React-хука, если DI избыточен.
- **Query-менеджер** → `Api` — реактивный, поэтому к нему можно подключить любой пользовательский код, а его параметры могут быть сами сигналами.
- **Dev-tools** → есть готовые DevTools на основе Redux DevTools для просмотра сигналов и query-состояний ([devtools](https://github.com/fozy-labs/rx-toolkit/blob/main/docs/devtools/README.md)).
- **Точка контакта с React** — это `useSignal`/`useScope` и провайдер скопа; всё остальное остаётся фреймворк-агностичным, потому что сигналы — это `Observable`.

### Картина целиком

```ts
import { z } from 'zod';
import { injectable, inject } from '@fozy-labs/simplest-di';
import { Signal, LocalSignal } from '@fozy-labs/rx-toolkit';
import { useSignal } from '@fozy-labs/rx-toolkit/react';

// Определяем класс
@injectable("SINGLETON")
class UserSession {
   userId$ = Signal.state<string | null>(null);

   isLoggedIn$ = Signal.compute(() => !!this.userId$());

   login(userId: string) {
      this.userId$.set(userId);
   }

   get userId() {
      return this.userId$.peek();
   }

   get isLoggedIn() {
      return this.isLoggedIn$.peek();
   }
}

// Используем в другом классе
@injectable("SCOPED")
class PanelMode {

   private _session = inject(UserSession);

   // LocalSignal.create — сигнал, синхронизированный с localStorage,
   // с валидацией через zod.
   mode$ = LocalSignal.create({
      key: "sidepanel-mode",
      userId: this._session.userId$, // Храним по юзер-специфичному ключу
      zodSchema: z.enum(['minimized', 'expanded']),
      defaultValue: 'minimized', // Тип для TS выводится из zod-схемы
   });

   toggle() {
      this.mode$.update(mode => mode === 'minimized' ? 'expanded' : 'minimized');
   }
}

// Используем в React-компоненте
function AppHeader() {
   const session = inject(UserSession);
   const isLoggedIn = useSignal(session.isLoggedIn$);

   return (
      <header>
         {isLoggedIn ? <LogoutButton /> : <LoginButton />}
      </header>
   );
}
```
